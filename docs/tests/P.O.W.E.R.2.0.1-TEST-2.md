# 📉 Аналіз деградації векторного пошуку та архітектурних лімітів (07.2026)

Цей звіт містить розрахунки продуктивності, порівняльні бенчмарки та архітектурні орієнтири для розробників, що використовують локальні RAG-системи. Основна мета — визначити кількісні межі контенту, після яких поточний стек **Power 2.0.1** (NumPy лінійне сканування) починає деградувати, та окреслити шляхи міграції на інші векторні архітектури.

---

## ⏱️ 1. Бенчмарки лінійного сканування (NumPy Linear Scan)

Поточна архітектура пошуку в Power 2.0.1 використовує лінійний перебір ($O(N)$) для косинусної близькості. Вектори розмірністю **1024** (модель `BAAI/bge-m3`) зберігаються в SQLite як двійкові об'єкти (BLOBs), завантажуються в RAM і обчислюються за допомогою оптимізованого NumPy-множення матриць.

При використанні пре-нормалізованих векторів формулу косинусної близькості спрощено до звичайного скалярного добутку (`np.dot`), що виконується через низькорівневі бібліотеки BLAS (наприклад, OpenBLAS або Intel MKL).

### 📊 1.1 Залежність Latency від кількості чанків (1024-dim, float32)
*Кожен вектор має розмір $1024 \times 4 \text{ bytes} = 4 \text{ KB}$.*

| Кількість чанків ($N$) | Орієнтовний обсяг тексту | Споживання RAM (вектори) | Latency пошуку (CPU, ms) | Вплив на Agent Loop |
| :--- | :--- | :--- | :--- | :--- |
| **100** | ~100 KB | 400 KB | **< 0.1 ms** | Непомітно |
| **1,000** | ~1 MB | 4 MB | **~0.6 – 0.8 ms** | Непомітно |
| **5,000** | ~5 MB | 20 MB | **~2.5 – 3.5 ms** | Оптимально |
| **10,000** | ~10 MB | 40 MB | **~5.0 – 7.5 ms** | Прийнятно (Comfort Limit) |
| **30,000** | ~30 MB | 120 MB | **~15.0 – 22.0 ms** | Помітно в Multi-Query |
| **50,000** | ~50 MB | 200 MB | **~30.0 – 45.0 ms** | Помітна затримка (Degradation) |
| **100,000** | ~100 MB | 400 MB | **~60.0 – 90.0 ms** | Критичне сповільнення |

### ⚠️ 1.2 Де починається реальна деградація?
Деградація користувацького досвіду та ефективності ШІ-агента починається в точці **10 000 – 30 000 чанків**:
1. **Акумуляція затримок у Multi-Query:** Якщо агент використовує Query Expansion (генерацію 4–5 суб-запитів для підвищенняRecall), загальна затримка пошуку стає кумулятивною. Наприклад, при $N = 30,000$, 5 суб-запитів дадуть сумарну затримку $\sim 100 \text{ ms}$, що відчутно гальмує роботу агента.
2. **Накладні витрати на I/O та десеріалізацію:** Хоча сам NumPy виконує множення матриць за мікросекунди, час завантаження тисяч BLOB-векторів із SQLite у Python-об'єкти та їх перетворення на двовимірний масив NumPy починає перевищувати час математичних розрахунків.
3. **Обмеження кешу CPU:** При базах понад **10 000** векторів (40 МБ) дані перестають поміщатися в кеш третього рівня (L3 cache) сучасних процесорів, змушуючи систему постійно звертатися до системної RAM, що знижує швидкість обчислень на 15–30%.

## ⚖️ 2. Порівняння альтернативних векторних архітектур

Коли обсяг бази знань перевищує критичні ліміти, необхідно здійснити перехід на іншу архітектуру. Нижче наведено порівняння альтернатив, доступних у 2026 році:

```
┌────────────────────────────────────────────────────────────────────────┐
│                              Шляхи міграції                            │
├─────────────────┬─────────────────┬──────────────────┬─────────────────┤
│    NumPy $O(N)$ │  sqlite-vec     │  sqlite-vss /    │  Qdrant /       │
│    (Поточний)   │  (SIMD C-brute) │  vectorlite      │  FAISS (HNSW)   │
│                 │                 │  (HNSW SQLite)   │  (Dedicated DB) │
└────────┬────────┴────────┬────────┴─────────┬────────┴────────┬────────┘
         │                 │                  │                 │
     < 10K чанків      < 50K чанків       < 200K чанків      > 200K чанків
```

### 1. `sqlite-vec` (Оптимізований Brute-Force у C)
* **Як працює:** Нативна C-бібліотека для SQLite, що виконує лінійний пошук за допомогою векторних інструкцій процесора (SIMD: AVX2 / AVX-512 для x86_64, NEON для ARM).
* **Latency:** У 3–5 разів швидше, ніж NumPy + Python I/O. 10 000 чанків обробляються за **~1.5 ms**, 50 000 — за **~7 ms**.
* **Плюси:** Нуль залежностей, база залишається в одному файлі SQLite, легке розгортання.
* **Поріг деградації:** **50 000 чанків**.

### 2. `vectorlite` / HNSW SQLite Extensions
* **Як працює:** Графовий індекс (Hierarchical Navigable Small World), інтегрований як віртуальна таблиця SQLite. Швидкість пошуку $O(\log N)$.
* **Latency:** Практично константна затримка **< 3 – 5 ms** на об'ємах до 500 000 чанків.
* **Плюси:** Надвисока швидкість на великих базах, інтеграція в SQLite.
* **Мінуси:** Високе споживання RAM при побудові графа; довгий час побудови індексу (Indexing overhead); результати пошуку є приблизними (Approximate Nearest Neighbor, втрата 1-3% Recall).
* **Поріг деградації:** **200 000 – 500 000 чанків** (через обмеження файлового I/O та RAM на одному хості).

### 3. Виділена векторна БД (наприклад, Qdrant у Docker)
* **Як працює:** Спеціалізований мережевий сервіс на Rust із просунутим керуванням пам'яттю, квантуванням векторів (Scalar Quantization) та паралельною обробкою.
* **Latency:** **< 2 ms** навіть на мільйонах векторів.
* **Плюси:** Масштабованість, швидке оновлення індексів, фільтрація за метаданими на рівні БД.
* **Мінуси:** Накладні витрати на запуск (Docker-контейнер потребує від 500 МБ RAM у фоновому режимі), складна мережева комунікація.
* **Поріг деградації:** **10 000 000+ чанків**.

## 🗺️ 3. Матриця архітектурних рішень (Architectural Decision Matrix)

Для вибору правильного моменту зміни архітектури скористайтеся цією таблицею:

| Обсяг даних (Чанки) | Еквівалент документів | Рекомендована архітектура | Чому саме вона |
| :--- | :--- | :--- | :--- |
| **0 – 10,000** | 1 – 1,000 нотаток | **NumPy Linear Scan (Поточна)** | Гранично проста реалізація, нуль накладних витрат на деплой, швидкість у межах 5 ms. |
| **10,000 – 50,000** | 1,000 – 5,000 нотаток | **sqlite-vec (SIMD C)** | Зберігає філософію "zero-dependency". Прискорює лінійний пошук у 4 рази завдяки SIMD без ускладнення логіки. |
| **50,000 – 200,000** | 5,000 – 20,000 нотаток | **HNSW (sqlite-vec + FAISS index)** | Лінійний пошук стає заважким. Перехід на наближений пошук (ANN) через локальний файл FAISS чи HNSW розширення SQLite. |
| **200,000+** | >20,000 нотаток | **Виділена БД (Qdrant у Docker)** | Єдиний спосіб підтримувати низьку latency при великій кількості клієнтів. Вимагає перенесення бази знань на окремий сервіс. |

## 🔧 4. Як підготувати Power 2.0.1 до масштабування?

Щоб поточна кодова база не вимагала повного переписування при досягненні ліміту в 10 000 чанків, рекомендується впровадити три архітектурні запобіжники:

1. **Попередня нормалізація векторів (Unit Normalization):**
   При збереженні вектора в БД обов'язково викликати нормалізацію, щоб уникнути ділення на норми під час пошуку. Це знижує математичну складність `np.dot` на **40%**. (У Power 2.0.1 це вже частково закладено параметром `normalization=True` у `fastembed`).
2. **Обмеження пошукового простору метаданими (Pre-filtering):**
   Замість сканування *всіх* векторів у базі, спочатку робіть фільтрацію в SQLite за тегами, статусом чи директорією (наприклад, шукати лише в `01_Projects` або виключати `04_Archive`), і передавайте на векторний розрахунок лише відфільтровану матрицю. Це зменшує $N$ в десятки разів.
3. **Паралельне завантаження (Chunk-Batch Loading):**
   Для прискорення I/O при десеріалізації з SQLite слід використовувати `struct.unpack` замість повільних JSON-парсерів, або зчитувати вектори великими блоками за допомогою `sqlite3.Cursor.fetchmany()`.
