## 1. Переаналіз історичного корпусу (`docs/tests/`)

Перечитано всі 13 попередніх звітів. Повторювані, остаточно не вирішені теми:

- **FP-6 (нестабільність GT)** — різні звіти використовували різну ground truth, що давало
  суперечливі висновки («FTS — найгірший» проти «FTS — ідеальний»). `2.2.2-TEST-2` спробував
  заморозити GT у `tests/fixtures/search_gt.json`. **Цей звіт доводить, що заморожений GT
  досі дефектний (B5).**
- **FP-3 / FP-4** — MiniLM-L12 (384d) сліпий до точних назв та UA↔EN перефразування.
  Вирішено у v2.2.3 перемиканням за замовчуванням на Qwen3 (1024d); **заблоковано на цьому хості через B3.**
- **FP-5** — reranker `ms-marco-MiniLM` шкодить мультимовному recall. У v2.2.3 постачається
  Qwen3-Reranker; **B6 показує, що reranker досі обвалює точність до 0 на практиці.**
- **FP-7** — _тихі_ порожні результати (гірше за краш). Ядро B1/B2.
- **FP-1 / FP-2** — розширення запиту / нормалізація; закрито у v2.1.0.

Найважливіший методологічний урок: **кожен попередній висновок «пошук хороший/поганий»
настільки ж надійний, наскільки надійна його GT.** B5 анулює числові базові лінії
у `2.0.1`–`2.2.2`.

## 3. Тестовий харнес

`bench_223.py` (закомічено) проганяє всі 5 режимів по фіксованому GT і обчислює
**MRR, Recall@5, nDCG@10, UDCG@10**. Фоновий RAM-вартовий опитує
`psutil.virtual_memory().used` кожні 1 с і **жорстко вбиває процес при >14 ГБ**.
Vault було синхронізовано один раз (`power sync --force`) перед read-only прогоном бенчмарку.

### 3.1 Знайдені та виправлені баги (B1–B4)

**B1 — тихий порожній семантичний пошук (`searcher.py:_semantic_search`)**
Коли `chunk_embeddings` був порожнім, старий код запускав _неблокуючий_ `request_sync`
і повертав `[]`. Запит до існуючого документа повертав ніщо — тихий FP-7.
_Виправлення:_ виконуємо **синхронну** пакетну синхронізацію ембедінгу, щоб запит повертав справжні результати.

**B2 — тихе пропускання через застарілий індекс (`searcher.py:_sync_vault_to_db`)**
Інкрементальна синхронізація шанувала кеш `mtime`, навіть коли таблиці ембедінгів були порожні
(звичне після синхронізації, що впала під OOM). Результат: 512 файлів проіндексовано, 0 векторів, семантичний
пошук назавжди порожній. _Виправлення:_ якщо `sync_embeddings` запитано, але `chunk_embeddings`
порожній, поки `file_metadata` має рядки — примусово ре-ембедимо все.

**B3 — алокація 97 ГБ у Qwen3-ONNX (`embeddings.py`)**
На цьому хості `qwen3_embed` запитував буфер **104 737 671 424 байт** (~97.5 ГБ) для
одного вузла MatMul і не зміг алокувати навіть при `batch_size=1`. Старий код потім тихо
видавав 0 векторів. _Виправлення:_ probe-embed при завантаженні моделі; при `RuntimeError`
виставляємо in-process прапорець `_QWEN3_DISABLED` і назавжди відступаємо на fastembed. Користувач
все одержує працюючий пошук замість порожнього індексу.

**B4 — гонка «database is locked» (`searcher.py:_semantic_search`)**
Синхронний ре-ембед відкривав друге з'єднання-письменник, поки з'єднання-читач було
ще відкрите → `database is locked` під навантаженням, втрата ембедінгів посеред синхронізації. _Виправлення:_ закриваємо
читача, запускаємо письменник на власному з'єднанні з busy_timeout 60 с, потім
відкриваємо читача знову.

**Виправлення стійкості тестів:** `tests/test_embeddings.py` стверджував `len(vec) == EMBEDDING_DIM`
(1024, обчислено для qwen3 за замовчуванням), що ламається, коли рантайм відступає на
MiniLM (384). Менеджери тепер відкривають авторитетну властивість `.dimension`; тест її використовує.

## 5. Коренева причина: ground truth хибний (B5 / FP-6)

Ми grep-перевірили очікування GT проти живого vault:

| Очікування GT                                                                   | Реальність у vault                                                                                                                                    |
| ------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------- |
| `MASTER-LESSONS-LEARNED.md` u=1.0 для «GPG signing commit»                      | файл містить **0** влучань «gpg signing»; справжній документ — `04_Archive/2026-04-24_GPG_Signing_Fix_and_Sync.md` (FTS правильно ранжує його **#1**) |
| `Server_Inventory.md` u=1.0 для «ZFS storage pool»                              | файл містить **0** «zfs»; ZFS-контент живе у `Successor-Hub.md` / нотатках обладнання                                                                 |
| `P.O.W.E.R.2.0.md` u=1.0 для «rendering mermaid graph»                          | файл **справді** містить «mermaid» ×4 — проте семантичний пошук досі не видає його (B6)                                                               |
| `POWER_v2.3+_Detailed_Implementation_Plan.md` u=1.0 для «embedding model UA/EN» | файл **справді** містить «embedding» ×14 — проте не знайдено (B6)                                                                                     |

**Висновок:** заморожений GT кодує застарілі/хибні очікування. Половина його
«правильних відповідей» насправді не містять термінів запиту. Це означає, що **всі попередні числові
висновки, побудовані на цьому GT (включно з FP-аналізом `2.2.2-TEST-2`), не верифіковані.**
GT треба регенерувати LLM-суддею, який _читає кандидат-документи_, а не вручну.

## 6. Справжнє обмеження якості retrieval (B6)

Навіть із правильним GT (документи mermaid/embedding, що _дійсно_ містять терміни), семантичний пошук
не ставить їх у top-50. Перевірено:

- `P.O.W.E.R.2.0.md` має **13 проембеджених чанків** (підтверджено в `chunk_embeddings`).
- Запит `"P.O.W.E.R. 2.0 mermaid graph related notes"` → top-10 — _інші_ mermaid-документи,
  `P.O.W.E.R.2.0.md` відсутній.
- Те саме для `POWER_v2.3+_Detailed_Implementation_Plan.md` (embedding ×14) на UA/EN
  запиті про ембедінг.

Ймовірні причини (дослідити у 2.2.4):

1. **Фіксований `SemanticChunker`** погано розбиває mermaid/код-блоки; релевантне речення
   потрапляє у чанк із низькою подібністю. Допоможе late-chunking / parent-child.
2. **MiniLM 384d** слабкий на цьому змішаному UA/EN технічному корпусі (FP-3/FP-4). Бажаний
   фікс Qwen3-1024d **заблоковано через B3** на хостах із обмеженою RAM.
3. Немає **multi-query expansion** — один перефразований запит недостатньо досліджує простір.

## 8. Рекомендації (для 2.2.4+)

1. **Регенерувати GT за допомогою LLM-судді**, який читає кандидат-документи; списати рукописний
   `search_gt.json` (B5). Доти всі історичні числа якості вважати неверифікованими.
2. **Вимкнути / заштрихувати reranker** за замовчуванням або перетренувати/розкалібрувати його для UA/EN; поточний
   `hybrid_reranked` суто негативний (B6, підтверджує `2.0.3-TEST-4`).
3. **Перейти на late-chunking / parent-child** замість фіксованих вікон (B6).
4. **Додати multi-query expansion** (+~12% Recall@20 за практикою 07.2026).
5. **Зробити Qwen3-ONNX безпечним на хостах із малою RAM**: передавати опції сесії ONNXRuntime
   (`enable_cpu_mem_arena=False`, `arena_extend_strategy=kSameAsRequested`) або постачати
   q4/q8 квантований ONNX-варіант, що алокує <2 ГБ (B3).
6. **Зберегти коректний відступ + захисти тихого порожнього результату** (B1–B4) — вони вже на місці.
